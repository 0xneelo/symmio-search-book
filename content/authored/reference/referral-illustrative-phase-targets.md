# Referral Illustrative Phase Targets

Neelo's product-metrics source treats phase targets as illustrative. That word matters. The page says targets should be finalized in public release notes for each milestone.

The compendium should preserve that boundary. It can explain what a phase is trying to prove without turning draft bands into a product promise.

## What Each Phase Tries To Prove

Pre-beta should prove controlled growth with stable operations and no critical flow outages. In this phase, the strongest signal is not maximum scale; it is whether the system can move through onboarding, attribution, listing, monitoring, and support without breaking.

Beta should prove measurable referral attribution, stronger listing velocity, and improved market activation rates. In this phase, the dashboard needs enough volume of behavior to test whether referral attribution connects to real market creation.

Open phase should prove broad participation with consistent quality controls and predictable conversion funnels. In this phase, the system has to preserve integrity while access is no longer tightly constrained.

## Release-Note Rule

The source's safest publication rule is simple: phase targets belong in release notes for the milestone that owns them.

That keeps targets time-bound and reviewable. It also prevents old target bands from becoming stale docs that users read as current guarantees.

## Publication Boundary

This page explains the illustrative target framework. It does not publish final target numbers, launch commitments, uptime guarantees, referral benefit schedules, listing quotas, quality thresholds, or payout formulas.
