# Referral Program Source Table Of Contents

The referral source is not a single policy page. It is a chaptered architecture package.

The overview states the thesis: referrals evolve from invite-gated onboarding into an incentive-aligned growth layer that can connect identity, rakeback, points, partner economics, and operational safeguards.

The table of contents matters because each chapter owns a different kind of claim. Baseline pages describe the current referral, rakeback, and points structure. Architecture pages define boundaries and claim flow. Rakeback, points, economics, access phasing, LP/category routing, security, metrics, open decisions, rewards packs, and tokenized-points scenarios each answer a different reader question.

## How To Read The Source Map

Use the source map as a routing layer:

- system baseline for current-state language;
- architecture for on-chain/off-chain and claim-flow boundaries;
- rakeback design for two-channel economics and tier policy;
- points and rewards for integrity, finalization, and utility;
- referral economics for anti-gaming and code policy;
- access phasing for pre-beta, beta, and open participation;
- LP and category layer for pool referrals and partner overlays;
- security controls for cooldowns, monitoring, and trust boundaries;
- metrics framework for KPIs and reporting standards;
- open decisions for unresolved policy;
- rewards packs and artifacts for future game-layer and TGE-accounting concepts;
- tokenized-points scenarios for explicitly hypothetical composability.

## Documentation Rule

When a user asks a referral question, route to the chapter that owns the claim instead of answering from a convenient neighboring page. This avoids using a metric page as policy, a rewards-pack page as live product promise, or a hypothetical tokenized-points page as current roadmap.
